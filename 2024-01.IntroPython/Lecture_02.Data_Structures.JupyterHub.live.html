1.0.0 Python data structures have properties and functions¶

As discussed in Lecture 01, everything in Python is an object:

  • integers
  • floats
  • strings

The above are all objects but also data types in Python. We can combine these data types in data structures to properly store, format, and model our data. A note on simulating "random" events: The only random things you'll find in computer science are whether your programs will run on the first try and whether you'll understand them 6 months from now.

More seriously, the ability to generate a truly random sequence of numbers is not possible. Usually a random number generator might use something like the system time as a seed to initialize its state but if we use a specified seed, we can get repeatable results.

1.7.3.3 Referencing an object versus copying it¶

Before we can continue we need to get our backup copy of the original gene_sequence which was stored in gene_sequence_copy. There are a few ways to go about doing this:

  1. Initiating a new list with list(). This also works with objects like dict and set which we'll learn about later.
  2. Using index notation [:] which returns a shallow copy!
  3. Importing the copy package and use the copy() or ...
  4. deepcopy() functions.
Are you making a copy or pointing at the original?: We have inadvertantly come to a very important point in Python about objects and memory. When we use the assignment statement object_X = object_Y, we are actually just asking Python "Where is object_X located in memory?" and what returns is a reference to that location that is assigned to object_Y. That means we are not copying the object but rather just pointing to it using a different variable! In some cases, this can be problematic because alterations to object_X will also be seen in object_Y and vice versa. Instead what we want is to duplicate (or copy) the information in object_X and store it in a new place in memory that is referenced by object_Y! That's where the above functions will help you out. The first three methods return a shallow copy (only the first layer of objects is duplicated) and the fourth will return a fully duplicated and independent copy of the object. Learn more about the difference between shallow and deep copies here.

2.0.0 Dictionaries are associative arrays¶
Dictionaries consist of keys that point to values (elements) in an unordered manner. Elements can even be more dictionaries!

So far, we have seen that lists are very useful to store diverse types of data in a single structure. However, lists are not that convenient to use when we need to extract data in groups or elements that "belong together". Let's revisit our nested list list_aminoacids from section 1.11.0. What if you were interested in getting any data associated with aspartic acid? Do you remember what index Asp was at? Under this circumstance, we are better off using dictionaries.

Dictionaries are similar to lists and they can also take (almost) any data type. Unlike lists, however, dictionaries are not ordered. Instead, dictionaries map a key to a set of values that are related to that key, i.e. keys:values that belong together. In our example, Aspartic acid is the key and an associated value is "Asp".

Dictionaries are created by using { } (curly brackets), a feature that also makes them easily identifiable. Unlike lists, though, dictionary keys are immutable and have no defined order. As a side note, Python dictionaries are usually called hashes in other programming languages. In summary:

  • Dictionaries use a key:value system.
  • Two entries cannot share the same key.
  • Multiple keys can have the same value.
  • Keys must be immutable data types (string, integer, or tuples, but NOT lists).
  • Values have few restrictions on data type.
  • Dictionaries themselves are mutable ie they can be added to, removed from, and changed.

3.0.0 Introduction to NumPy¶

Built-in Python objects are very versatile. They can hold different data types, you can do some basic mathematical operations with them, they can be mutable or not, and many more. However, they lack a key feature: performing simple and advanced mathematical operations on data structures, in ways that are time- and computationally-efficient.

NumPy (Numeric Python) is a package for scientific computing developed by Travis Oliphant and first released in 2005, based on a pre-existing python package called Numeric. NumPy has advanced and changed a lot since its first release, all thanks to a very active community of programmers that have contributed their time and effort to improve NumPy.

In terms of data structures, NumPy offers an alternative to built-in lists called NumPy arrays, which allow math operations across one- or multi-dimensional arrays. 3.6.1.1 Using the axis parameter to calculate the mean across dimensions¶

As you can see from above, when we use the default behaviour of mean, it treats our array, regardless of dimension, like a flat list of numbers and calculates the mean of the whole set. What if, we want to calculate mean across rows or columns? Intuitively it can be a little confusing but recall that we use [row, column] notation to access arrays. Another way to think about it is that axis=0 returns a row of means, and axis=1 returns a column of means.

4.4.4 Is it a View or a Copy?¶

Okay, it appears that we can use the [ ] to do almost anything but beware! We've been playing around a lot by pulling out sub-portions of our data frame. There is a lot happening under the hood but remember that we are dealing with objects! Depending on how we ask a Pandas object for access to it's data, it may be returning a view of the object or a copy!

In the above example we used .loc[row, col] notation to access our DataFrame but we also used .loc[row][col]. We've seen this before in accessing arrays as well, the ability to subset two ways.

In the first case, we are multi-indexing by calling on a method and passing two parameters, row and column, to the method. For Python and the DataFrame object, it all happens in a single step by essentially going to that reference and pulling out what we want using the DataFrame's internal methods.

In the second case, we are chain-indexing. Depending on a package's implementation this can give you very different results! In the case of a DataFrame, we are asking Python to first retrieve just the rows of the DataFrame we want. When that object is returned it needs to be assigned a temporary place in Python memory. At this point, it could become a separate object or entity from the original - a shallow copy! After that, we are then subsetting by col in a completely different command (by Python's viewpoint). If we were using these commands to set values in our DataFrame we could be setting them in an object that simply disappears!
Remember to try and use multi-indexing when possible. Chain-indexing may only bring sorrow.

Simply put, unless you have good reason to, with Pandas objects such as a DataFrame, avoid chain-index notation!

Read more: For some deeper examples of this problem, check out the Pandas documentation on this specific issue.

6.1.0 What happens when we slice a list?¶

Here's a small example of list slicing and what we get back. Remember we talked about getting a view or copy back in section 4.5.4?

Let's recall that every object is assigned an integer ID in Python. You can find this with the id() function. When we assign a variable to an object, Python is providing a link or reference back to the ID of that object. Python essentially uses the IDs to assign memory space to an object where it's value can be stored. Python uses references to keep track of IDs so that when there are no more (0) references to an object ID, Python can reclaim the memory where its value is heald and its ID to use for something else in the program.

Whenever we slice through a list, it will return a copy of the references to those elements. Rather than copying each element to a new space in memory (and object ID), it uses the reference to find where those individual objects are in memory. This can bring up some rather tricky concepts when we have nested lists that are made using other list object references and what happens to them.

Depending on how you have sliced your original list, you may be getting back a direct reference (ID) to the original object, or references for the elements of the original list!

6.1.1 Assigning and slicing our lists¶

  1. list_1 = ["genome", 20, 30.5, True]: we generate a list with 4 elements.

  2. list_2 = [list_1, "this", "that", "those"]: we generate a second list of 4 elements with the first element being a list itself

  3. list_3 = list_2[0:3]: make a list copying the first 3 references to list_2; this includes the reference to the list_1 object.

  4. list_4 = list_1[0:4]: make a list by copying references for the first 4 elements in list_1; we do not copy the reference to list_1 object itself.

  5. list_5 = list_2[0][0:4]: make a list by copying references for the first 4 elements in the first element of list_2; it should be very similar to list_4.

  6. list_6 = list_2[0]: copy the reference to the first element of list_2 which is the reference to the list_1 object.

  7. list_7 = list_2[:]: copy all of the references to the elements of list_2 which includes a reference to the list_1 object.

  8. list_8 = list_2.copy(): Use the list.copy() method to create a shallow copy of the contents of list_2 which includes a reference to the list_1 object.

  9. list_9 = copy.copy(list_2): Use the copy.copy() function to create a shallow copy of list_2 which includes a reference to the list_1 object.

  10. list_10 = copy.deepcopy(list_2): Use the copy.deepcopy() function to create a deep copy of list_2 which will duplicate new versions of all elements, including making a copy in memory of list_1!

6.1.2 Make a change directly to list_1¶

  1. list_1[0] = "genomic": Now we've changed the first element to list_1. Any other objects directly referencing list_1 will propogate this change. Objects that merely reference the elements of list_1 will not propagate this change.

6.2.1 Slicing an array with [ ] returns a reference to the original array¶

So, the way arrays handle slicing is different to how lists handle the same commands. From the simple example above, we see that the underlying behaviour is that when you slice an array with [] you are always returned a reference or view of that array. That being said, altering an array slice's reference can be quirky.

  1. array_4[0,0] = 50 is a direct slicing call with an assignment and so the change is propagated to array_1 and array_2

  2. array_3 = array_3 * 2 does not use any slicing notation. Instead the Python interpreter makes a new copy of array_3, completes the math on the array and then assigns this to the variable array_3. The reference that array_3 had to array_1 is released and replaced with one for this new object.

These are intentional behaviours of the array object. If a reference to an array is not what you want, then you can use the .copy() method to generate a shallow copy of the array. Any changes in the original will not be propagated to the copy and vice versa!
